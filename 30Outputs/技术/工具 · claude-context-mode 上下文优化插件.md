# claude-context-mode - Claude Code 上下文优化插件

> 仓库：https://github.com/mksglu/claude-context-mode

## 一句话概括

在 Claude Code 和工具输出之间加一层「沙箱 + 检索」缓冲带，把大输出变成可检索资产，按需取片段，减少上下文占用。

---

## 问题背景

很多人用 Claude Code 一段时间后都会遇到同一个问题：**模型不是不聪明，而是上下文先被工具输出吃空了。**

一次 Playwright 快照、一次 GitHub issue 列表、一次日志读取，都可能把几十 KB 原始数据直接灌进上下文窗口。几轮下来，回答质量就开始下滑，后面的推理和连续性也会明显变差。

**真正的矛盾是**：你需要更多工具来解决复杂问题，但工具又在持续挤占上下文。

这个项目的价值在于：**不是减少工具调用，而是改变工具输出进入上下文的方式。**

---

## ⚠️ 潜在问题

| 风险 | 说明 |
|------|------|
| **上下文不完整** | 只拿片段可能漏掉关键信息，引发错误推断 |
| **检索词偏差** | 查询词不够精确，拿回来的内容就跑偏 |
| **幻觉风险** | 基于不完整信息做推断，可能产生错误结论 |
| **推理链断裂** | 有些看似不相关的信息可能是关键证据 |

本质上是用「精确度换容量」，检索不精准就会出问题。

---

## ✅ 适用场景

| 场景 | 说明 |
|------|------|
| 日志分析 | 日志量大，人工也看不完 |
| 信息密度低的大输出 | API 回包、测试输出等 |
| 关键词明确 | 已知要找什么 |
| 重度用户 | 已遇到过上下文爆炸问题，需要精细控制 |

---

## 🚫 不适用场景

| 场景 | 原因 |
|------|------|
| 代码审查 | 需要全局视角 |
| 探索性问题 | 还不知道要找什么 |
| 高密度关键内容 | 合约、协议、核心代码不能漏 |
| 刚上手 Claude Code | 还没遇到上下文瓶颈 |

## 工作原理

### 架构概览

Claude Code 发起工具调用 → Context Mode 负责路由 → 执行类请求进入沙箱，内容类请求进入 FTS 索引 → 高相关片段回流到对话上下文

### 三步流程

1. **先在沙箱里执行和处理数据**
2. **大输出优先入索引，不直接塞进对话**
3. **按意图或查询词回传最相关片段**

原始数据不再一股脑冲进上下文，而是转成「可检索资产」。需要哪部分，再精确拿哪部分。

---

## 核心能力详解

### 1. execute / execute_file：把执行留在沙箱里

命令和文件处理都走隔离子进程。默认回流的是处理后的 stdout，而不是整坨原始数据。

对日志分析、测试输出、API 回包这类「体积大但信息密度不均」的场景，这个策略非常实用。

### 2. index / search：把大内容变成可检索知识库

用 SQLite FTS5 做索引，配 BM25 排序，并结合词干处理。

这不是「截断前 N 个字符」，而是围绕命中词返回相关窗口。你得到的是可用证据片段，不是随机前缀。

### 3. batch_execute：把多轮工具调用压成一轮

支持「多命令执行 + 多查询检索」一次完成。

对调研型任务尤其关键：如果本来要执行 20 次命令再查 10 次，batch 化能明显减少往返和上下文噪音。

### 4. fetch_and_index：网页抓取后直接入检索

网页先转 markdown 再入库，后续按查询取片段。原始 HTML 不进上下文，减少无效占用。

---

## 亮点：对子代理的路由约束

在插件模式下，`hooks/pretooluse.mjs` 会对高风险路径做引导或拦截，推动子代理优先走 context-mode 工具链。

这点很关键。很多团队主代理很克制，但子代理会疯狂读写和抓取，最后还是把窗口打爆。

**把规则写进 hook，相当于把「最佳实践」变成「默认行为」。**

---

## 效果表现

README 和 BENCHMARK 里给了多个真实场景，方向很一致：

**原始数据体积和进入上下文的数据体积之间，通常存在明显数量级差距。**

带来的直接收益：同样工作量下，会话更耐久，模型在后半程仍保持稳定推理。

对经常做调试、日志分析、仓库研究的人，这个收益非常现实。

---

## 快速上手

1. 安装 marketplace 插件
2. 安装 `context-mode@claude-context-mode`
3. 重启 Claude Code
4. 跑一段日常任务后执行 `/context-mode:stats` 看节省情况

**建议做一个最小实验**：选一个平时会爆上下文的任务，分别在「启用 / 未启用」下跑到 10~15 轮，对比后半程回答质量和可追问深度。

## 作者声明

> 它解决的是「上下文资源分配」问题，不替代你的判断。

需要逐字审阅完整原文时，仍要拉原文；查询词过泛时，检索质量会下降。

---

## 总结

底层逻辑：把「该放在工作层的数据」留在沙箱和索引，把「该进入思考层的信息」按需送给模型。

这不是花哨优化，而是让 AI 开发流程从「短跑」变成「耐力赛」的关键工程化步骤。
